Cache-Related terms from RFC 2616 (HTTP/1.1)

  1. Cache
    A cache is a storage layer (browser, proxy, server) that saves copies of HTTP responses (like HTML, images, CSS). Purpose: reduce load, save bandwidth, and speed up response time by reusing stored responses instead of fetching from the origin server.

  2. Freshness
    A cached response is fresh if it is still allowed to be served without revalidating.
    Freshness is controlled by headers like:

    • Expires: "don't use after this time"
    • Cache-Control: max-age=SECONDS: "this response is fresh for SECONDS"
    If the cached response is still within this time window -> itâ€™s fresh.

  3. Validation / Revalidation
    When a cached response becomes stale, the cache must check with the origin server before using it again.
    This check uses validation headers:
    • ETag: unique ID of the version of a resource
    • If-None-Match: asks "Is this ETag still valid?"
    • Last-Modified
    • If-Modified-Since
    If the server replies 304 Not Modified, the cache can reuse the old stored version.

  4. Stale
    A stale response is one that is too old to be used directly.
    It has passed its expiration time (Expires / max-age).
    A cache must revalidate stale responses unless rules like
    Cache-Control: must-revalidate change the behavior.

  5. Cache-Control
    A powerful HTTP header that controls how caching works.
    Important directives include:
    • no-store: do not store under any situation
    • no-cache: store it, but always revalidate before using
    • public: any cache may store
    • private: only browser cache, not shared proxies
    • max-age: number of seconds it stays fresh
    • must-revalidate: cannot use stale responses without rechecking
